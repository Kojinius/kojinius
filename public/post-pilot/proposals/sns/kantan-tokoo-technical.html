簡単投稿プラン
Client-Side Draft Supply + Web Intent

サーバ常駐ゼロのクライアントサイド完結型。コアバリューは 「下書き供給機構 + Web Intent 投稿起動」。業種テンプレ × 季節カレンダー × LocalStorage 履歴で 3案生成、Notifications API でリマインド、X/Threads の Web Intent で公式アプリへ deep-link。OAuth・API キー・cron 一切不要。

Runtime: Static + Service Worker Auth: 不要 ¥0 / 月 X Intent (加重280) Threads Intent (500) Storage: LocalStorage Notification: Web Notifications API Verified: 2026-05-26
2026-05-28 改訂 — 対応チャネルは X / Threads の2媒体 本仕様書は当初 4 チャネル（X / Threads / LINE / Jimoty）想定で作成しましたが、現行の簡単投稿プランは X / Threads の2媒体に整合済みです。Jimoty は営業ツールとして手動活用（システム連携は未定・今後検討）、LINE は本プランでは現状外し（友だち配信は「盛り盛りMAXプラン」の LINE Messaging API へ）。各媒体は本文＋ハッシュタグの文字数上限（X=全角2換算の加重280／Threads=500）を超えると投稿ボタンが無効になります。以降の §03・§04 等に残る LINE / Jimoty の記述は、当時の設計根拠として参考掲載しています。

3案選定ロジック

type Draft = { id: string; tone: "polite"|"friendly"|"promo"; seasonTags: string[]; weight: number; text: string; hashtags: string[] };
type SeasonEvent = { date?: string; dayOfWeek?: number; tags: string[]; weight: number };

export function pickTodaysThree(
  templates: Draft[],
  seasonal: { events: SeasonEvent[]; recurring: SeasonEvent[] },
  history: { draftId: string; postedAt: number }[],
  now = new Date()
): Draft[] {
  // 1. 今日に該当する季節タグを集める
  const activeTags = new Set<string>();
  const today = now.toISOString().slice(0, 10);
  const dow = now.getDay();
  seasonal.events.filter(e => e.date === today).forEach(e => e.tags.forEach(t => activeTags.add(t)));
  seasonal.recurring.filter(e => e.dayOfWeek === dow).forEach(e => e.tags.forEach(t => activeTags.add(t)));

  // 2. 30日以内に投稿済みの draftId を除外
  const THIRTY_DAYS = 30 * 86400_000;
  const recent = new Set(history.filter(h => now.getTime() - h.postedAt < THIRTY_DAYS).map(h => h.draftId));
  const pool = templates.filter(d => !recent.has(d.id));

  // 3. 季節マッチでスコアリング
  const scored = pool.map(d => {
    const match = d.seasonTags.filter(t => activeTags.has(t)).length;
    return { d, score: d.weight * (1 + match * 0.7) };
  });

  // 4. トーン別に1件ずつ Top を選ぶ
  const pickTop = (tone: Draft["tone"]) =>
    scored.filter(s => s.d.tone === tone).sort((a, b) => b.score - a.score)[0]?.d;

  return [pickTop("polite"), pickTop("friendly"), pickTop("promo")].filter(Boolean) as Draft[];
}

LocalStorage 履歴管理

const KEY = "kt:posted:v1";
const MAX = 100;

export function loadHistory(): { draftId: string; postedAt: number; channel: "x"|"threads" }[] {
  try { return JSON.parse(localStorage.getItem(KEY) ?? "[]"); }
  catch { return []; }
}

export function recordPost(draftId: string, channel: "x"|"threads") {
  const hist = loadHistory();
  hist.unshift({ draftId, postedAt: Date.now(), channel });
  localStorage.setItem(KEY, JSON.stringify(hist.slice(0, MAX)));
}

const onClickLine = (draft: Draft) => {
  if (!isMobile()) {

    return;
  }

function withHashtags(d: Draft) {
  return d.hashtags?.length
    ? `${d.text}\n\n${d.hashtags.map(h => `#${h}`).join(" ")}`
    : d.text;
}

05通知リマインダの実装

Web Notifications API + Service Worker で、サーバなしのローカル通知を実現。

許可取得

export async function enableReminders(times: string[] /* ["07:00", "12:00", "21:00"] */) {
  if (!("Notification" in window)) throw new Error("unsupported");
  const perm = await Notification.requestPermission();
  if (perm !== "granted") return false;

  const reg = await navigator.serviceWorker.register("/sw.js");
  localStorage.setItem("kt:reminders", JSON.stringify(times));
  scheduleNext(times);
  return true;
}

再訪問時の catch up

// app/_app.tsx などで起動時に毎回
export function rescheduleOnVisit() {
  const times = JSON.parse(localStorage.getItem("kt:reminders") ?? "[]");
  if (!times.length) return;
  scheduleNext(times); // 次回発火を SW に再登録
}
iOS Safari の制約 iOS Safari は Notification.requestPermission() を Web ページから直接呼べない。PWA としてホーム画面追加された後の navigator.serviceWorker からのみ通知可能（iOS 16.4+）。Android Chrome / Desktop は標準対応。iOS 対応が必須なら PWA 化（manifest.json + apple-touch-icon）が前提。
setTimeout の永続性 Service Worker は idle で 30秒〜数分で停止する。長時間 setTimeout は信頼性が低い。実運用では「次回訪問時に未発火分を即発火 + 直近分のみ setTimeout を仕掛ける」ハイブリッドを採用。Periodic Background Sync (Chrome 限定) があれば優先利用する分岐を追加可。

07セキュリティ考慮

XSS / URL injection 下書きを動的に組み立てる場合は URLSearchParams または encodeURIComponent を必ず通す。生の文字列を + 連結する実装は禁止。
下書きデータの取り扱い テンプレ JSON に HTML / script を含めない運用ルールを保持。React の自動エスケープに任せ、dangerouslySetInnerHTML は禁止。
noopener / referrer policy window.open(url, "_blank", "noopener,noreferrer") を統一。reverse tabnabbing 対策。
機密情報の不在 API key・OAuth token・refresh token を一切持たない。漏洩リスクの根本がない。LocalStorage 履歴もユーザ端末完結でサーバ送信なし。

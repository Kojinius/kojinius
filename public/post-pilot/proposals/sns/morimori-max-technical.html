02技術スタック

既存資産（Vercel Pro / Notion / Chatwork）を再利用。新規追加はゼロ。

ランタイム / 配信基盤

RUNTIME

Vercel (Pro Plan)

Fluid Compute + Functions

Node.js 22 / Edge 不可（外部 API レイテンシ吸収のため Node Functions で 60〜300s 実行枠を確保）

選定理由: 既存 Pro 契約。Cron Jobs ネイティブ、Cold Start なし、env 暗号化標準。
SCHEDULER

Vercel Cron Jobs

vercel.json → crons[]

cron 式で /api/cron/* をトリガ。最小単位は 1分。Pro なら回数無制限。

選定理由: 別 VPS 不要、Vercel 内完結。GitHub Actions cron より発火精度が高い。
DATABASE

Notion DB

@notionhq/client

投稿キュー、配信履歴、業種テンプレを Notion で人間が編集可能に保持。

選定理由: 非エンジニアでも編集可。Firestore 化は流量増えてからで間に合う。

AI 生成

TEXT LLM

Claude Sonnet 4.6

@anthropic-ai/sdk

原稿生成・モデレーション・ハッシュタグ最適化を 1モデルで統合。Prompt Cache で system 部を再利用。

選定理由: Sonnet 4.6 は日本語コピー品質が高く、tool_use とキャッシュが安定。
IMAGE GEN

fal.ai FLUX.1 [schnell]

@fal-ai/client

1枚 約¥0.45、生成 ~2秒。商用ライセンスクリア。レスポンスは CDN URL を Notion に保存。

選定理由: Imagen より圧倒的に安く、品質も SNS 用途では十分。

配信チャネル（自動3種 + 営業ツール・性質が異なるため統一実装不可）

CHANNEL: X (公開投稿)

X API v2 (Pay-per-use)

POST /2/tweets · OAuth 2.0 PKCE

2026/02 以降は従量課金が新規デフォルト。テキスト ¥2.25、URL 含む ¥30、画像投稿は media/upload 別途。

注意: 旧 Basic (¥30,000/月) は新規受付停止。無料枠なし。x-rate-limit-* ヘッダで動的に制御。
CHANNEL: Threads (公開投稿)

Threads Graph API (Free)

graph.threads.net/v1.0

2段階フロー: POST /me/threads → 30秒待機 → POST /me/threads_publish。24h 内 250投稿まで。

注意: 課金なし。long-lived token は 60日有効、cron で 50日目に refresh。
CHANNEL: Jimoty (営業ツール)

Jimoty (システム連携は未定)

no API · 営業ツールとして手動活用

ジモティは地域掲示板として、現状はスタッフが手動で営業・集客に活用します。post-pilot システムからの連携（下書き自動生成・通知など）は未定（今後検討）です。

将来連携する場合の留意: 公式 API が無く、利用規約で「クロール等で自動的に収集・解析する行為」が禁止のため完全自動投稿は不可。連携する場合も AI 下書きを Chatwork 通知 → 人手投稿の半自動に限定し、非広告トーンで規約適合させる。
CHANNEL: LINE (Push配信)

LINE Messaging API

POST /v2/bot/message/push · Bearer Auth

2025/03 で廃止された LINE Notify の公式代替。X/Threads の「公開投稿」とは性質が異なる「友だち追加済ユーザへの Push 配信」。

料金 (2026/05時点): コミュニケーション ¥0 / 200通月、ライト ¥5,000 / 5,000通、スタンダード ¥15,000 / 30,000通。通数 = 配信回数 × 受信者数。2026/10 に料金改定予定（要再確認）。

03処理フロー（1ジョブの内訳）

Vercel Cron 1回のトリガで完了するステップ。idempotent 設計で再実行安全。

  1. キュー取得 Notion DB から status=ready かつ scheduled_at ≤ now のレコードを 1件 pop。
  2. 原稿生成 Claude Sonnet 4.6 に業種・トーン・テンプレを投げて投稿文＋ハッシュタグを JSON で受け取る。cache_control: ephemeral で system prompt をキャッシュ。
  3. 画像生成 fal.ai FLUX.1 schnell に原稿要約をプロンプトとして送信、CDN URL を取得。
  4. モデレーション Claude を再呼び出しし、PII / hate / brand-safety をスコアリング。閾値未満なら自動却下。
  5. 承認（任意） Chatwork に下書きを通知。承認モード時は Webhook 待機（最大15分）、auto-pass モード時はスキップ。
  6. X 投稿（公開） POST /2/tweets。画像ありなら先に media/upload。x-rate-limit-remaining を監視。
  7. Threads 投稿（公開） POST /me/threads でコンテナ作成 → 30秒スリープ → POST /me/threads_publish。失敗時は指数バックオフで 3回まで再試行。
  8. Jimoty（営業ツール・連携未定） ジモティは現状スタッフが手動で営業活用。システムからの下書き生成・通知の連携は未定（今後検討）。連携する場合も POST 自動化は行わず、Chatwork 通知 → 人手投稿の半自動に限定する。
  9. LINE Push 配信 条件を満たす場合のみ POST /v2/bot/message/push で対象セグメントへ配信。月の通数上限（プラン依存）を監視し、超過時はスキップ + Chatwork に警告。
  10. 記録 Notion DB に posted_at / x_tweet_id / threads_post_id / line_request_id / jimoty_handed_off_at を書き戻し、status を posted に更新。

06コード骨子

/api/cron/post-publish.ts — Vercel Cron から呼ばれるエントリ。

// vercel.json
{
  "crons": [
    { "path": "/api/cron/post-publish", "schedule": "0 7,12,21 * * *" },
    { "path": "/api/cron/refresh-tokens", "schedule": "0 3 */50 * *" }
  ]
}

// app/api/cron/post-publish/route.ts
import Anthropic from "@anthropic-ai/sdk";
import { fal } from "@fal-ai/client";
import { Client as Notion } from "@notionhq/client";

export const maxDuration = 300; // Threads publish 待ち含む

export async function GET(req: Request) {
  verifyCronSecret(req);
  const job = await popReadyJob(notion);
  if (!job) return Response.json({ skipped: true });

  const draft = await generateDraft(anthropic, job);
  const image = await fal.subscribe("fal-ai/flux/schnell", { input: { prompt: draft.imagePrompt } });
  const safety = await moderate(anthropic, draft.text);
  if (!safety.pass) return rejectAndLog(notion, job, safety);

  if (job.requireApproval) await awaitChatworkApproval(job, draft);

  const [xRes, thRes] = await Promise.allSettled([
    postToX(draft.text, image.url),
    postToThreads(draft.text, image.url),
  ]);

  await recordResult(notion, job, xRes, thRes);
  return Response.json({ ok: true, x: xRes.status, threads: thRes.status });
}

07運用上の注意点

X トークン管理 OAuth 2.0 PKCE の refresh_token は 6ヶ月で失効。cron で残期間を監視し、3ヶ月時点で更新フローを Chatwork に通知すること（ヘッドレスでの refresh は X 側で 2025 末に制限強化）。
Threads publish ラグ コンテナ作成から publish までは Meta 側で最低 30秒必要。maxDuration: 300 必須。連投時はキューを 1job = 1post で分割し、Cron の発火間隔を最低 1分空ける。
レート制限の防御 X は x-rate-limit-remaining ヘッダ、Threads は HTTP 429 で検知。指数バックオフ（1s → 5s → 30s）でリトライし、3回失敗で Notion に failed 記録 + Chatwork 通知。
URL 含む投稿のコスト跳ね X はリンク付き投稿が ¥30/件と通常の 13倍。URL は x_post_options.include_url=false で原則オフ、CV 計測したい時だけ明示的にオンにする。
Jimoty 連携時の留意（将来検討・現状は営業ツール） ジモティは現状スタッフが手動で営業活用し、システム連携は未定。将来連携する場合は、利用規約「投稿等をクロール等で自動的に収集、解析する行為」の禁止に加え、出品代行・商用広告色の強い投稿も規制対象である点に留意し、非広告トーン + 健康カテゴリ固定の方針を継承、「専用 Jimoty アカウントで運営者が手動投稿」のワークフローを守る。cron からの直接 HTTP POST は実装しない。
LINE 通数の事前見積もり 通数 = 配信回数 × 受信者数。友だち 1,000人に週1配信で月 4,000通 → ライト ¥5,000 圏内。友だち 5,000人へ月60投稿は 300,000通 → スタンダード超過。Notion DB に line_audience_size を持ち、cron 開始時に「今月の予想通数 + 残り枠」を試算して上限超過なら Chatwork に warning 通知 + Push をスキップする防御を入れる。
LINE Notify 廃止後の移行 旧 LINE Notify（〜2025/03/31）の置き換え。Messaging API は「友だち追加」フローが必須なので、エンドユーザに QR を提示する初期導入手順がクライアント側に発生する。営業時の説明資料に組み込み済み（プレゼン版参照）。

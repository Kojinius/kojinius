03既存技術スタック (meoscope)

2026-05-26 時点で開発中（本番確認可）の構成。meoscope/CLAUDE.md + 設計書 main.md §4 + 実コード裏取り済。

ランタイム / 認証

カスタムドメイン meoscope.com で本番稼働。VERCEL_PROJECT_PRODUCTION_URL 使用 (Deployment Protection 回避済)。

選定理由: 既存 Pro 契約。lienrapport プロジェクト群と Blob/Neon 共有。

React 19 · TypeScript

App Router で API Route + Server Component。serverExternalPackages で playwright-core / chromium-min を external 化。

AUTH

Firebase Authentication

Google Sign-In + Admin SDK

セッション Cookie 14 日 (createSessionCookie)。__session Cookie 名 · httpOnly · sameSite=lax · secure (production)。

選定理由: Kalenex 方式踏襲。Resend magic link は UX 不可で却下済。
EXIST

Neon ホワイトリスト

meoscope.users · meoscope.invites

email ユニーク + role (admin/member)。Firebase で認証後、SELECT role FROM meoscope.users WHERE email = $1 で許可判定。未登録は 401。

分析パイプライン

EXIST
CORE API

POST /api/analyze (SSE)

runtime: nodejs · maxDuration: 300

8 ステップを SSE で逐次送出: 対象特定 → 競合抽出 → 評判取得 → デジタル存在感 → 定量分析 → ポジション判定 → SWOT → AI 提案。途中失敗は step:warn、致命は step:error。

name モード注意: 候補 2 件以上で type:'candidates' を送って接続終了 → クライアント側で /select UI で選択 → 再 POST。完全自動化されない。
EXIST
GEO DATA

Google Places API + OSM

Places (新世代) · Overpass · Nominatim

Places は無料枠 1,000 Place Details / 月。Overpass で半径 3km の競合 POI 抽出。Nominatim で地名解決。

quota ガード: 利用量を meoscope.quota_counters に記録、閾値超で step:warn 返却。
EXIST
LLM

Anthropic Claude API

SWOT + Recommend (2 呼び/分析)

分析 1 回あたり 2 回の Claude 呼び出し。利用量は meoscope.anthropic_usage_log に自前記録 (Admin API は Individual account 不可)。

EXIST
CORPORATE

gBizINFO API + Web Fetch

業種推測 confidence='low' fallback

business type signals で confidence=low と判定された場合に runCorporateSubset が起動。GBIZ_API_KEY 投入済。SSRF 対策の fetchSafeHtml で公式サイト多段 fetch。

PDF / ストレージ

EXIST
PDF

Playwright + chromium-min

/dashboard/[id]/print + headless render

POST /api/report/pdf で reportId を受領 → /dashboard/[id]/print を headless ブラウザで開く → Google Fonts (Noto Sans JP) document.fonts.ready 待ち + 1s → page.pdf({ format: 'A4', margin: 12mm }) → Blob 保存 + stream 返却。

制約: Cookie ベース認証必須。担当のセッション Cookie を Playwright Context に渡してから render する。テンプレは マルチページの詳細版のみ。

04運用フロー (Phase A の 1 ジョブ内訳)

DM 受信から PDF 添付返信までの手動運用フロー。営業担当のステップだけを記述。

  1. DM 受信 (Instagram / ジモティ) 担当が DM に気付く。営業時間外なら翌営業日対応の自動返信文を 1 本入れておく運用が望ましい
  2. meoscope を開く https://meoscope.com · Google Sign-In · 初回のみホワイトリスト登録 (admin が meoscope.invites に行を追加 → 担当がリンク踏む)
  3. S1 分析開始画面で店舗名入力 name モード · 任意で areaHint (例: 小田原市) + categoryHint (例: 飲食店) を併記すると候補精度が上がる
  4. 候補が複数なら正しい店を選択 候補 1 件なら即採用 · 2 件以上は /select 画面で人手選択 → 再 POST · 候補ゼロならエリア併記して再試行
  5. 分析実行 (SSE 進捗を画面で見る) 8 ステップ進捗が SSE で流れる · 致命的エラーなら step:error 表示 · Claude が 2 回呼ばれるので体感数分 (実測未済・本番運用前に担当と一緒に計測する)
  6. S3 ダッシュボードで結果確認 「次の一手」「SWOT」「KPI」「9 チャート」を一通り目視確認 · 違和感あればこの段階で気付ける
  7. S4 PDF 出力 「PDF ダウンロード」ボタン押下 · POST /api/report/pdf · Playwright が /dashboard/[id]/print を A4 multi-page でレンダリング · ファイル名 meoscope-{reportId}.pdf
  8. PDF を DM に添付返信 Instagram DM は PDF 直添付不可 (画像/動画のみ) → 1 ページ目を PNG 化して画像添付 + 「フル版は別途お送りします」運用 · ジモティはメッセージから外部 URL OK · 詳細運用は次節 §05
  9. 商談 → 受注 → 動画制作 レポートの弱点を起点に提案 · 受注したら同じ PDF を納品時に「導入前レポート」として渡す
  10. 90 日後 (Phase A は手動カレンダー) 担当のカレンダーに納品から 90 日後リマインダを手動登録 · その日に meoscope で再分析 · 「動画前 → 動画後」の比較レポートを送付 · Phase B で自動化検討

05Instagram DM への PDF 添付制約

Instagram DM の仕様上、PDF 直添付ができない点は事前に詰めが必要。

媒体PDF 直添付外部 URL運用案
Instagram DM 不可 可 (プレビュー展開) PDF 1 ページ目を画像化して添付 + Blob の限定 URL を DM 本文に貼る
ジモティ メッセージ 仕様確認要 可 外部 URL 経由が無難 · ジモティ規約「商用色 / クロール禁止」に抵触しない範囲で
メール (info@ 等) 可 可 チラシ 3 の MAIL 欄経由のフローはメール添付で完結
限定 URL の作り方 (Phase A 範囲) 既存 uploadPdf は Vercel Blob (Public ストア) に PUT しているが、現行実装では URL を外に出さず Function 経由で stream するだけ。担当が外部 URL として共有したい場合は、/api/report/pdf-share のような 短期 URL 発行ルートを新規追加するか、PDF を Google Drive に手動アップロードして Drive のリンクを貼るのが現実的。最初は Drive 経由で運用し、頻度が高くなったら API 化を判断。

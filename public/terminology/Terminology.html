RBAC

Role-Based Access Control

アールバック／ロールベースアクセス制御

セキュリティ2026-07-02 追加

ユーザー一人ひとりに権限を直接くっつけるのではなく、まず「ロール（役割）」に権限をまとめておき、ユーザーにはロールだけを割り当てる方式のアクセス制御。「この人は編集できる？」ではなく「この操作は manager 以上」と役割で考えるのがポイント。

人が増えたり異動したりしてもロールを付け替えるだけで済み、権限の付け忘れ・消し忘れが起きにくい。逆にユーザー個別に権限を配ると、退職・異動のたびに全機能を棚卸しするはめになる。

用例Kalenex2026-07-02

4段階のロール（admin / manager / staff / viewer）を数値の階層にして、hasMinRole(userRole, minRole) ひとつで「manager 以上か？」を判定している。同じロール階層をファイル読み取りAPIの認可（authorize-file-read.ts）や Firestore セキュリティルール（per-task RBAC / per-node RBAC）でも使い、判定基準がクライアント・サーバ・ルールでズレないようにしている。

export type Role = 'admin' | 'manager' | 'staff' | 'viewer';

export const ROLE_HIERARCHY: Record<Role, number> = {
  admin: 4, manager: 3, staff: 2, viewer: 1,
};

export function hasMinRole(userRole: Role | null, minRole: Role): boolean {
  if (!userRole) return false;
  return ROLE_HIERARCHY[userRole] >= ROLE_HIERARCHY[minRole];
}

backfill

バックフィル（埋め戻し）

ばっくふぃる

データベース2026-07-02 追加

新しいフィールドや仕組みを導入したとき、既存のデータにも同じ値を一括で書き込んで「追いつかせる」作業。新規データはコードが自動で正しく作るが、過去のデータは自分で埋めない限り古いまま──この差を埋めるのが backfill。

本番データを書き換えるので事故ると重い。だから「dry-run（お試し）→ 件数を確認して承認 → apply（本書き込み）→ backup を残す」という段取りで行うのが鉄則。

fallthrough

セキュリティ2026-07-02 追加

rollback

ロールバック（巻き戻し）

ろーるばっく

パスワードの代わりに、メールで届く1回限りのログイン用リンクをクリックして本人確認する認証方式（パスワードレス認証の一種）。パスワードを覚えさせない・漏らさない利点がある。

OAuth

Open Authorization

おーおーす

認証・権限2026-07-02 追加

「パスワードを渡さずに、他のサービスへ限定的な許可を与える」ための標準規格。「Google でログイン」「カレンダーへのアクセスを許可しますか？」画面の裏側がこれ。渡されるのはパスワードではなくトークン（合鍵）で、範囲（scope）と期限があり、あとから取り消せる。

dist
